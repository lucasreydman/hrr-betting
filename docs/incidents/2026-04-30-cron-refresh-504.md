# 2026-04-30 вЂ” Cron `slate-refresh` 504s on `/api/refresh`

## What failed

GitHub Actions workflow `Cron вЂ” sim/lock/settle`, job `slate-refresh`,
calling `POST /api/refresh`. Three failed runs in the last 24h:

| Time (UTC) | Run ID | SHA |
|---|---|---|
| 2026-04-30 09:35:55 | 25158308338 | `3e7546c` |
| 2026-04-30 07:32:47 | 25153220894 | `3e7546c` |
| 2026-04-29 22:11:08 | 25136527948 | `3e7546c` |

Failure mode in every case:

```
curl: (22) The requested URL returned error: 504
##[error]Process completed with exit code 22.
```

`504 Gateway Timeout` from Vercel = the function exceeded
`maxDuration = 10` (Hobby plan cap) before responding.

## Root cause

Pre-existing intermittent timeout on `/api/refresh`. The endpoint runs
`rankPicks(date)`, whose per-game block was a **serial** `for (const
game of games)` loop. On a fully-loaded slate, each game's per-batter
fan-out + Supabase cache reads cost ~300вЂ“700ms. Sum across 11 games on
a slow upstream tick = >10s, triggering the 504.

Failures predate commit `4904f7b`. Not a regression introduced by recent
work вЂ” but commit `4904f7b` (just before this incident note) makes the
ranker also process **finalised** games (was `if (final) continue`). On
a slate with 9 finals + 2 lives, that adds another ~4.5s of serial
work, which would have made these timeouts dramatically more frequent.

## Why I noticed before it got worse

The commit that made the ranker process finalised games (`4904f7b`)
landed minutes before the GitHub failure notification. None of the 30
most recent runs in `gh run list` had run on `4904f7b` yet вЂ” every
recent failure was on `3e7546c`. The fix below ships before any
`4904f7b` cron tick has a chance to amplify the existing flakiness.

## Files changed

- `lib/ranker.ts` вЂ” replaced serial `for (const game of games)` with
  `await Promise.all(games.filter(...).map(async game => {...}))`. Each
  iteration accumulates picks into local `r1` / `r2` / `r3` arrays and
  returns them; the outer code merges into the rung accumulators after
  the parallel block resolves. Wall-clock for the per-game phase drops
  from `sum(per-game)` to `max(per-game)`.

## How it was fixed

Parallelisation only вЂ” no caching changes, no schedule changes, no
auth changes, no data-shape changes.

The boxscore live-settle loop after the per-game phase was already
parallel (`Promise.all` over distinct final game IDs); left untouched.
